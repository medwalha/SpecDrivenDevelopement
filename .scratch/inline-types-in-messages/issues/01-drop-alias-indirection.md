# Drop the `alias` indirection — message fields use primitive types directly

> *This was generated by AI during triage.*

## Reporter intent

> "I don't want to add aliases under types — I need to use them directly in messages."

The current `/pdf-to-cpp-types` + `/pdf-to-cpp-messages` workflow forces a `using Isin = char[12]` style alias to exist in `include/messages/types/Aliases.hpp` before a message struct can declare a field of that shape. The reporter wants message structs to declare those fields directly (`char m_SecurityID[12];`, `int64 m_MsgSeqNum;`), with no alias layer in between.

The canonical example `examples/PostTrade.hpp` already uses primitive types directly (`char m_SecurityID[12]`, `int64 m_MsgSeqNum`) — so this change **realigns the prompts with the example**, rather than introducing a new style.

## Agent Brief

**Category:** enhancement
**Summary:** Eliminate the `alias` type category. Message structs declare scalar/ASCII-array fields directly using the typedefs from `Common.hpp` and raw `char[N]`. `Aliases.hpp` is removed; `/pdf-to-cpp-types` only handles `enum`, `bitmap`, and `struct` categories.

### Current behavior

- `/pdf-to-cpp-types` accepts four categories: `alias`, `enum`, `bitmap`, `struct` — each routed to a dedicated consolidated header (`Aliases.hpp` / `Enums.hpp` / `Bitmaps.hpp` / `Structs.hpp`) under `include/messages/types/`.
- `/pdf-to-cpp-messages` enforces a hard rule (the "Règle d'or") that forbids declaring a field as `char m_Foo[N]` "if a dedicated alias exists" — e.g. it requires `Isin m_SecurityID;` instead of `char m_SecurityID[12];`. When a referenced alias is missing the prompt **stops generation** and tells the user to run `/pdf-to-cpp-types` to create it first.
- The reusable-type table in `/pdf-to-cpp-messages` lists `Aliases.hpp` as one of four sources to map fields against.
- The aggregator `include/messages/types/Types.hpp` lists `Aliases.hpp` among its includes (per the `/pdf-to-cpp-types` skeleton specification; the on-disk `Types.hpp` is stale and references a legacy `GeoPosition.hpp` instead — fix this opportunistically as part of this work).
- `examples/PostTrade.hpp` **does not follow the alias rule** — it uses `char m_SecurityID[12]`, `char m_Currency[3]`, etc., directly. This is a contradiction between prompts and canonical example.

### Desired behavior

- `/pdf-to-cpp-types` accepts only three categories: `enum`, `bitmap`, `struct`. The `alias` category is removed from documentation, variable enum lists, skeleton specifications, and the "règles strictes" section.
- `/pdf-to-cpp-messages` instructs message structs to declare fields directly:
  - Scalar fields → typedefs from `Common.hpp` (`uint8`, `int16`, `int64`, …).
  - Fixed-width ASCII strings → raw `char m_Field[N];`.
  - Composed types (`Decimal`, `TimestampNano`, `RptGrpHeader`, …) → still referenced by name from `types/Structs.hpp`.
  - Enums and bitmaps → field still typed as the underlying integer (`uint8`/`uint16`); the `enum class` / mask constants from `types/Enums.hpp` / `types/Bitmaps.hpp` remain available for *semantic* validation only, unchanged from today.
- The "Règle d'or" in `/pdf-to-cpp-messages` no longer forbids `char m_Foo[N]`. The blocking flow ("stop the message and run `/pdf-to-cpp-types`") only triggers when an `enum`, `bitmap`, or `struct` is missing — never for a missing alias.
- `include/messages/types/Aliases.hpp` is removed if present (and ensured absent if not). The aggregator `include/messages/types/Types.hpp` is rewritten to include exactly the three remaining category headers (`Enums.hpp`, `Bitmaps.hpp`, `Structs.hpp`) and **no other files** — explicitly no `GeoPosition.hpp`, no `Aliases.hpp`.
- `examples/PostTrade.hpp` continues to compile and remains the canonical reference (no edits required, but verify after the prompt changes).

- **`Decimal`, `TimestampNano`, `RptGrpHeader`, and any other composed struct** in `Structs.hpp` are **not** to be inlined — they remain named types referenced from message fields.
- **`MessageHeader` / `PacketHeader` workflow** (the étape 0 "blocking prerequisites" gate) is untouched.

- **Any change to `examples/PostTrade.hpp`** — it is the validation reference; if it stops compiling after the changes, the changes are wrong, not the example.
